# tree-sitter-phits

[![Version](https://img.shields.io/badge/version-0.1.0-blue.svg)](https://github.com/alsciaukat/tree-sitter-phits/releases)
[![License: MIT](https://img.shields.io/badge/license-MIT-green.svg)](#license)

A [tree-sitter](https://tree-sitter.github.io/) grammar for the input files
(`.inp`) of **PHITS** (Particle and Heavy Ion Transport code System).

It parses PHITS input into a concrete syntax tree — sections (`[Material]`,
`[Cell]`, `[Surface]`, `[T-Track]`, …), MCNP-style cell geometry, material and
surface definitions, parameter/tally statements, and the region-keyed data
tables (`[Volume]`, `[Weight Window]`, `[Forced Collisions]`, …) — for use in
editor highlighting, structural navigation, linting, and other tooling.

> **Status: `0.1.0`.** Early release. The core sections parse cleanly; a few
> sections (`[Transform]`, `[Magnetic Field]`, `[Data Max]`, `[Frag Data]`, …)
> are still handled by a generic line body rather than a dedicated grammar.
> Node names and field names may change before `1.0.0`.

## Requirements

- [tree-sitter CLI](https://github.com/tree-sitter/tree-sitter) `>= 0.25`
  (`cargo install tree-sitter-cli`, `npm install -g tree-sitter-cli`, or your
  package manager)
- A C compiler (to build the shared library)
- [Node.js](https://nodejs.org/) `>= 18` — only if you want the Node bindings
  or `npm test`

## Install

Clone the repository:

```sh
git clone https://github.com/alsciaukat/tree-sitter-phits.git
cd tree-sitter-phits
```

The generated parser lives under `src/` and is committed, so no code
generation is needed just to parse. If you edit the grammar, regenerate it:

```sh
tree-sitter generate
```

### Node bindings (optional)

```sh
npm install          # builds the native addon via node-gyp-build
npm test             # runs the binding tests
```

## Run

### Parse a file

```sh
tree-sitter parse path/to/input.inp        # print the syntax tree
tree-sitter parse -q path/to/input.inp     # quiet: timing + first error only
```

The command exits non-zero if the tree contains any `ERROR` or `MISSING`
nodes. Try it against the bundled examples:

```sh
tree-sitter parse test/phits.inp
```

### Interactive playground

```sh
npm start            # builds the WASM grammar and opens the web playground
```

### Highlight the whole test corpus

Every file under `test/` is expected to parse with zero errors:

```sh
for f in test/*.inp; do
  echo -n "$(basename "$f"): "
  tree-sitter parse "$f" 2>&1 | grep -cE 'ERROR|MISSING'
done
```

## Editor integration

### Emacs

`phits.el` provides a `phits-inp-ts-mode` major mode built on Emacs' native
tree-sitter support (Emacs 29+). It expects the compiled grammar at
`~/.emacs.d/tree-sitter/libtree-sitter-phits.so`. Build and deploy it with:

```sh
./commands.sh build          # tree-sitter generate + build the shared library
```

Then load `phits.el` and open a `.inp` file:

```elisp
(add-to-list 'load-path "/path/to/tree-sitter-phits")
(require 'phits)
(add-to-list 'auto-mode-alist '("\\.inp\\'" . phits-inp-ts-mode))
```

### Other editors

The grammar ships bindings for C, Rust, Go, Node, Python, and Swift (see
`tree-sitter.json`). Any editor with tree-sitter support can consume the
grammar built from this repository.

## Development

`grammar.js` is the **only** hand-edited source; everything under `src/` is
regenerated by `tree-sitter generate`. `commands.sh` wraps the common loop:

```sh
./commands.sh parse <name>   # generate, then parse test/<name>.inp,
                             #   writing the tree to test/<name>.out and a
                             #   lexer/parser debug trace to test/<name>.debug
./commands.sh build <name>   # generate + build the shared library for Emacs
```

The bundled `manualE-phits.pdf` is the PHITS reference manual — consult it to
understand what a construct means before extending the grammar. See
[`CLAUDE.md`](CLAUDE.md) for a detailed architecture and rule reference.
